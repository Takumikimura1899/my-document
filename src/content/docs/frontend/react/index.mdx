## 対象範囲

- 状態管理・副作用・パフォーマンス最適化など、React 特有の検討事項
- コンポーネントの構造化・テスト・アクセシビリティへの配慮

## 開発の基本姿勢

- UI 実装は `src/components`（仮）など共通の配置ルールに従い、再利用性と可観測性を高める
- Hooks で状態や副作用を分離し、テスト可能なカスタム Hooks を積極的に活用する
- ブラウザ固有 API に依存する場合はフォールバックや SSR 時の挙動に注意する

## 推奨トピック

- コンポーネント命名とフォルダ構造
- 状態管理と副作用の整理
- パフォーマンス最適化
- テストとアクセシビリティ

## React コンポーネント構造

- 1 ファイル 1 コンポーネントを基本とし、関連するスタイル・テストを同階層に配置する
- props の型定義は `Props` インターフェースで明示し、`React.FC` での暗黙的な children 受け取りは避ける
- レンダリングロジックとユーティリティを分離し、複雑な処理はカスタム Hooks へ抽出する

## 状態管理と副作用

- UI 状態は `useState`/`useReducer`、ビジネスロジックはカスタム Hooks や外部ストア（例: Zustand）を検討する
- 非同期処理は `useEffect` だけでなく、`react-query` などのデータフェッチライブラリの活用も検討する
- サーバーコンポーネントとの連携時は境界を明確にし、クライアント側の処理は `use client` ディレクティブで管理する

## パフォーマンス最適化

- レンダリング回数を抑えるため、`React.memo` や `useMemo`/`useCallback` を適切に利用する
- リストレンダリングでは `key` を安定させ、仮想リストが必要か評価する
- `useTransition` や `Suspense` を活用し、UX の滑らかさとレスポンスを両立する

## テストとアクセシビリティ

- React Testing Library を用い、ユーザー視点の挙動をテストする
- Storybook で状態やバリエーションをドキュメント化し、デザイナーとの共通フイードバックループを構築する
- aria 属性・ロール付与・フォーカスマネジメントを徹底し、アクセシビリティ基準を満たす
