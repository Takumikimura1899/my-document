## このドキュメントの目的

- 新規バックエンドサービスやマイクロサービス立ち上げ時の技術スタック選定を支援する
- チーム内で共通の評価軸を持ち、議論の前提を合わせる
- 既存システムの見直しやリプレイス検討時の観点を提供する

## 評価軸

| 観点           | 具体例                                               | 重要度の目安       |
| -------------- | ---------------------------------------------------- | ------------------ |
| プロダクト特性 | レイテンシ要求、スループット、バッチ/ストリーミング  | 中〜高             |
| チーム体制     | 開発者の経験、学習コスト、採用のしやすさ             | 高                 |
| エコシステム   | フレームワーク・ライブラリ・ミドルウェア連携の成熟度 | 中                 |
| 運用性         | デプロイ/監視ツールとの親和性、運用自動化のしやすさ  | 中〜高             |
| 生産性         | 言語機能、型安全性、テスト容易性                     | 高                 |
| パフォーマンス | 起動時間、メモリ使用量、並行処理性能                 | プロダクト要件次第 |

> ※ 重要度は ⬆︎ に行くほど一般的に重視しがちな順。プロジェクトに応じて重み付けを調整してください。

## TypeScript（Node.js ランタイム）

キーワード: スピード & 一貫性

フロントエンドとバックエンドで言語と型定義を共有することで、圧倒的な開発スピードと一貫性を実現します。

### 強み

- Next.js / Astro などフロントエンドと共通言語化でき、フルスタック体制との相性が良い
- Vercel / Cloudflare Workers などサーバーレス環境へのデプロイが簡単
- TypeScript による型安全性と IDE サポートで初期学習コストが低い

### 選定しやすい状況

- 小〜中規模の API や BFF（Backend for Frontend）で、リードタイムを短くしたい
- フロントエンドチームが主体や兼任でバックエンドを担当する
- サーバーレス / エッジ実行など起動速度や柔軟性を重視する

### 注意点

- マルチスレッド並列が苦手（ワーカー/キューを活用する運用設計が必要）
- 長期運用では Node.js の LTS サイクルに合わせたアップデート対応が必須
- CPU バウンド処理には向かず、別サービスやワーカーを切り出す判断が必要

## Go

キーワード: シンプル & パフォーマンス

言語仕様がシンプルでありながら、非常に高いパフォーマンスを発揮します。マイクロサービスアーキテクチャに適しています。

### 強み

- シンプルな言語仕様と組み込みの並行処理（goroutine）で高スループットを実現しやすい
- 単一バイナリデプロイでコンテナ化・運用が容易
- 高速なコンパイルと低いランタイムオーバーヘッドでリソース効率が良い
- 標準ライブラリで HTTP、gRPC、テストなどが充実

### 選定しやすい状況

- 高いパフォーマンスと多くの同時接続が要求されるAPIサーバー
- マイクロサービスアーキテクチャのサービスの一つとして、特定の機能に特化したAPIを開発する場合
- フレームワークに依存せず、標準ライブラリ中心のシンプルな実装をチームで推進できる場合

### 注意点

- ジェネリクスの表現力は限定的で、複雑なドメインロジックで抽象化が難しい場合がある
- エコシステムはシンプル志向で、フルスタックなフレームワークは少ない
- Error handling や struct のバリデーションなどに冗長なコードが生じやすい

## Spring Boot（Java）

キーワード: 堅牢性 & エコシステム

大規模で複雑なアプリケーションを開発するための、実績豊富で成熟したエコシステムを持っています。

### 強み

- 大規模エンタープライズ開発で蓄積されたベストプラクティスと豊富なライブラリ
- DI コンテナ、バッチ、セキュリティ、監視などエコシステムが統合的に提供される
- JVM のパフォーマンスとマルチスレッドサポート、成熟した GC チューニング
- マイクロサービス構成での Observability、Config Server、Service Mesh 連携なども整備
- 開発者が豊富で、採用市場も安定している(ただし一定レベル以上のスキルを持った人材は限られる)

### 選定しやすい状況

- 大規模チームでの長期運用や、複雑なドメインとトランザクション管理が必要
- 既存 Java エコシステムやオンプレ / マルチクラウドへの統合が求められる
- エンタープライズ基盤（Spring Cloud、Spring Batch、Spring Security等）を活用したい

### 注意点

- 起動時間・メモリ使用量が比較的高く、軽量な関数実行やエッジ用途には不向き
- フレームワーク機能が豊富な分、学習範囲と設定項目が多い
- JVM バージョン管理とセキュリティアップデートのオペレーションが必要
- 開発環境のセットアップやビルド時間が長くなりがち

## 選定プロセスのすすめ方

1. **ビジネス要件を整理する**: SLA/SLO、トラフィック規模、拡張予定、デプロイ先制約
2. **チーム成熟度を評価する**: 既存のスキルセット、教育コスト、採用市場
3. **運用設計を検討する**: CI/CD、モニタリング、インフラ運用の自動化方針
4. **PoC / Spike を実施する**: 実装・デプロイ・監視の一連を試し、フィードバックを得る
5. **ADR に記録する**: 選定理由とトレードオフを決定記録に残し、定期的に見直す

## まとめ

| 技術        | 得意領域                                       | 主な決め手                                       |
| ----------- | ---------------------------------------------- | ------------------------------------------------ |
| TypeScript  | BFF、小〜中規模サービス、サーバーレス          | フロント連携、自動化された DX、軽量さ            |
| Go          | 高性能 API、ストリーミング、クラウドネイティブ | 並行処理性能、単一バイナリ、運用シンプルさ       |
| Spring Boot | 大規模・複雑ドメイン、エンタープライズ統合     | 豊富なエコシステム、ガバナンス、長期運用の安定性 |

必要に応じて複数技術を併用し、境界ごとに最適なスタックを選ぶ方針も検討してください。
