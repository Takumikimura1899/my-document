## 基本原則

- **可読性優先**: 同じ意図を表現する場合は明確さを優先し、コメントや命名で背景を補足する
- **小さな変更単位**: プルリクエストは 500 行以内を目安とし、段階的にレビューしやすくする
- **自動化の活用**: フォーマッタ・リンタ・テストを必ず通し、手動確認だけに依存しない

## 設計・実装のルール

- エラーハンドリングは失敗ケースを明示し、ログ・ユーザー通知の責務を切り分ける
- Feature Flag や設定値は環境変数・設定ファイルで管理し、ハードコードを避ける
- 依存追加時は「なぜ必要か」「代替は検討したか」を PR 説明欄に記載する

## レビューガイドライン

1. 変更概要・確認方法が説明されているか
2. 自動テストや手動確認が十分か（必要であればチェックリストを含める）
3. ドキュメント更新が必要かどうか確認し、漏れがあれば合わせて更新する

## 命名と構造

- ファイル名・ディレクトリ名は英語で統一し、ケバブケース (`my-component.ts`) を基本とする
- コンポーネントやユーティリティのエクスポートは名前付きエクスポートを優先する
- テストファイルは `*.test.ts` / `*.spec.ts` などターゲットに合わせた命名で配置する

## ドキュメントへの反映

- コーディング規約に変更があった場合はこのページを更新し、Slack の開発チャンネルで共有する
- 大きなルール変更は振り返りで合意をとり、適用開始日や対象範囲を明確にする
- 設計原則に関わる変更は [SOLID 原則](/docs/essentials/solid/) も合わせて更新する
